Architecture Narrative

Skymark follows a three-layer architecture: a frontend built with Vite + TypeScript using CodeMirror 6 for the editor and a custom preview controller with DOMParser-based rendering; a pure Rust core library (skymark-core) that handles Markdown parsing, HTML generation, sanitization, and smart-edit logic — designed to compile to wasm32; and a Tauri 2 backend (skymark-app) that exposes commands for file I/O, vault scanning, file watching, and draft management.

The frontend communicates with the backend exclusively through Tauri's invoke/listen mechanism. The render pipeline is a single canonical path: markdown text → pulldown-cmark → HTML → ammonia sanitization → return to frontend → DOMParser → replaceChildren → KaTeX/Mermaid/Prism enrichment. The vault mode adds a file tree sidebar, multi-tab editing, fuzzy search, broken-link detection, and a file watcher for external-change detection. Draft-based crash recovery saves periodic snapshots to app_data_dir.

Architecture

Strength
Separation of Concerns — skymark-core is pure

skymark-core has zero dependencies on Tauri, browser APIs, or any non-WASM-compatible crates. This cleanly separates the Markdown rendering logic (parsing, AST, sanitization, smart-edit) from the host shell. The crate can be compiled to wasm32-unknown-unknown, enabling a future web app with minimal rework.

Strength
Single Canonical Render Pipeline

The live preview and HTML export share the exact same code path (render_html → sanitize). There is no drift between what the user sees while editing and what gets exported. This eliminates a class of "preview looks different from export" bugs.

Strength
Atomic File Writes

Both save_file and export_file use write-temp-then-rename, preventing partial writes from corrupting user documents. Draft files also use atomic writes. This is a reliable-pattern for local-first apps.

Strength
Explicit Save Model with Draft Recovery

The app never silently saves — users control when files are persisted. Drafts are saved every 10 seconds (or on idle) with atomic writes, and the recovery system detects conflicts with external file changes, surfacing a 3-way prompt. This is a well-thought-out crash-recovery strategy.

Concern
God Module — skymark-app/src/main.rs

main.rs (90 lines) handles Tauri builder setup, plugin initialization, menu construction (with nested prepend logic), watcher state management, menu event routing, and invoke_handler registration. This single file is the entry point, the app configuration, and the command dispatch router. Extracting menu construction into a separate module would reduce coupling.

Concern
God Module — frontend/src/main.ts

The frontend main.ts (~400 lines) is the central hub for: file flow, draft handling, vault management, tab management, heading indexing, link checking, tree rendering, palette, toolbar, export dropdown, update banner, keyboard shortcuts, menu events, watcher events, sidebar resize, draft recovery, and startup restoration. No single file should orchestrate all of these concerns. This makes it hard to reason about, test, or modify any one concern in isolation.

Concern
Missing Abstraction Layer — Direct std::fs in Tauri Commands

Tauri command handlers in commands.rs and draft.rs call std::fs::read_to_string, std::fs::write, and std::fs::rename directly. There is no storage abstraction trait. This makes it impossible to swap in a different storage backend (e.g., encrypted storage, virtual file system) and couples the command layer to the concrete file system.

Concern
No Centralized State Management in Frontend

The frontend has no centralized state store. State is distributed across module instances (files, drafts, vault, tabs, headings, links, tree, palette) plus localStorage for vault root and sidebar width. There is no single source of truth, making it difficult to track the complete app state or implement undo/redo across the entire application.

Risk
Frontend Has No Error Boundary for Preview

If the preview pane crashes (e.g., KaTeX throws on malformed math), the error is caught at the invoke level and a text node is inserted into the preview. However, the editor keeps accepting input, and there is no mechanism to recover the preview state. A full re-render of the current content is needed but not automatically triggered.

Risk
Vault Scan Has No Depth Limit

The scan_dir function uses recursion with no depth cap. On a vault with deeply nested directories (e.g., node_modules-like structures), this could cause a stack overflow. The 5000-file cap prevents unbounded work but does not prevent deep traversal before hitting the cap.

Security

Strength
HTML Sanitization — Allowlist with Strict Attribute Control

The sanitizer uses ammonia with a carefully curated allowlist of tags, per-tag attribute restrictions, and generic_attributes for data-line. Event handlers (on*), script/iframe/object/embed elements, and javascript:/data: URL schemes are all stripped by default and never relaxed. This is a strong defense against XSS in markdown content.

Strength
DOMParser-Based Preview Rendering (Never innerHTML)

The preview controller uses DOMParser to parse HTML into a detached document, then imports nodes into the live DOM via replaceChildren. This prevents any embedded scripts from executing, providing a second layer of defense beyond the ammonia sanitizer.

Strength
Path Validation on All File Operations

Every Tauri command that touches the file system validates that the path is absolute and has an allowed extension (.md, .markdown, .txt for source files). Relative paths and arbitrary extensions are rejected. Draft keys are validated against path traversal (no /, \, ..). This is a solid defense against path traversal attacks.

Strength
Tauri Capabilities — Deny-by-Default

The capability model is deny-by-default. Each Tauri command the frontend may invoke is named explicitly. File-system scope is dynamic — granted only for the active vault root and its attachments folder, never the whole disk. This follows the principle of least privilege.

Strength
Local-First Design Collapses Attack Surface

As a local-first desktop app, Skymark has no network server, no user database, no authentication layer, and no external API calls (except optional auto-update). The OWASP attack surface is significantly reduced — the primary trust boundary is the markdown content itself, which is sanitized at render time.

Concern
File Watcher Accepts Paths from Frontend Without Additional Validation

The watch_paths command receives file paths from the frontend (via tabs.entries.map) and passes them directly to notify_debouncer_mini. While the paths originate from the vault scan (which validates abs_path), there is no additional validation at the watcher boundary. If a future feature introduces arbitrary user input to the watch list, it could be used to watch arbitrary system directories.

Concern
Draft System — FNV-1a Hash for Keys Is Not Cryptographic

Draft keys are generated using FNV-1a, which is fast and deterministic but not collision-resistant. While not a direct security risk in a local-first app, it means two different file paths could theoretically produce the same key (extremely unlikely in practice). More importantly, the draft content path stored in DraftMeta.original_path is used as-is without normalization, which could lead to inconsistent key generation if the same file is opened via different path representations (symlinks, relative vs absolute).

Concern
No Rate Limiting on Auto-Save Drafts

The draft system saves every 10 seconds or on idle ≥1s. There is no rate limit on how frequently the frontend can trigger draft saves. A malfunctioning editor or rapid tab switching could cause excessive file I/O, though the atomic write pattern prevents data corruption.

Risk
No Encrypted Storage for Sensitive Data

Settings are persisted as plain JSON in app_data_dir. While Skymark doesn't currently store credentials, if future features add API keys or sync tokens, they would be stored in plaintext. The architecture has no secrets management pattern in place.

Scalability

Strength
50ms Render Debounce Budget Is Well-Defined

The architecture explicitly budgets 50ms for preview updates and implements request ID deduplication to drop stale render results. This is a measurable performance target that guides development decisions.

Strength
Diff Rendering Strategy for Large Documents

The design spec defines a per-block diff render approach using BlockId (structural hash) to map AST blocks to DOM elements. Only changed blocks are re-rendered, and enrichment (KaTeX/Mermaid/Prism) is limited to changed regions. This is the right approach for maintaining the 50ms budget on large documents.

Concern
No Render Result Caching

Every keystroke that survives the debounce window triggers a full re-parse of the entire document. There is no content-hash-based caching layer. For documents with many small edits (e.g., typing one character at a time), the entire Markdown → AST → HTML pipeline runs repeatedly on nearly-identical input. A cache keyed on content hash would skip re-parsing for unchanged content ranges.

Concern
Vault File Watcher Re-Watches on Every Tab Change

The watchCurrentTabs function calls invoke("watch_paths", { paths: tabs.entries.map(...) }) every time a tab is opened or closed. This creates a new debouncer and drops the old one. There is no connection pooling or incremental watch management. On a vault with many open tabs, this could cause brief periods where file changes are not detected during the watcher swap.

Concern
Vault Scan Is Synchronous and Blocks

The scan_vault command runs a recursive directory walk on the Tauri backend thread. For a vault with thousands of files, this could block the Tauri event loop briefly. The 5000-file cap limits worst-case work but the traversal is still synchronous. A background task with progress reporting would be more scalable.

Risk
No Virtualized Preview for Large Documents

The preview renders all blocks in the DOM simultaneously. For a 50k+ word document with many blocks, this creates a large DOM tree that slows down scroll-sync resolution (which queries all [data-line] elements on every scroll event) and increases memory usage. The design spec acknowledges this as an out-of-scope concern but it should be flagged as a hard ceiling.

Reliability

Strength
Robust Draft-Based Crash Recovery

The draft system is a strong reliability feature. Drafts are saved periodically (every 10s or on idle), written atomically, and the recovery system detects external file changes to avoid overwriting user edits made outside the app. The 3-way prompt (keep draft / keep disk / diff) gives users control. Draft GC after 30 days prevents unbounded disk growth.

Strength
500ms File Watcher Debounce

The notify_debouncer_mini debouncer applies a 500ms delay before emitting file-change events. This prevents rapid file system events (common on macOS with FSEvents) from flooding the UI with reload prompts. The debounce is applied consistently across all watched files.

Strength
Atomic Write Pattern Prevents Data Corruption

All file writes (save_file, export_file, draft saves) use the write-temp-then-rename pattern. This ensures that a crash mid-write never leaves a partially-written file on disk. The temp files are prefixed with "." so they are hidden from typical file browsers and won't appear in the vault file tree.

Concern
Draft System Has No Race Condition Protection

There is no locking or file-level mutex around draft writes. If the auto-save timer fires while the user is also explicitly saving (Cmd+S), both operations could race on the same draft file. While atomic writes prevent corruption, the draft's saved_at_unix timestamp could be overwritten by a slightly newer save, causing the recovery system to lose track of which version is most recent.

Concern
File Watcher Has No Retry Logic

If watch_paths fails (e.g., the file is deleted between vault scan and watch registration), the error is silently swallowed with a toast. There is no retry mechanism. If the watcher process crashes or the file system becomes temporarily unavailable, there is no automatic recovery — the user must manually re-open the vault to re-establish watching.

Concern
Auto-Update Is Fire-and-Forget

The update check runs 3 seconds after launch with no timeout or retry. If the GitHub API is unreachable or the network is down, no update check occurs. There is no fallback (e.g., checking a mirror URL) and no user-visible error message beyond "You're up to date" when the check fails silently.

Risk
No Structured Logging or Error Reporting

The application uses console.error for error reporting in the frontend and Result types with to_string() error conversion in Rust. There is no structured logging system, no error aggregation, and no way to diagnose issues in production. For a local-first app this is less critical than for a cloud service, but it makes debugging hard for both users and developers.

Revised File Watcher Architecture

Incremental watch management with a persistent WatcherManager that adds/removes individual file watches without recreating the entire debouncer.

Actionable Recommendations

  1. Extract menu construction from main.rs — Move the File and Edit menu item creation into a dedicated menu.rs module. This reduces main.rs from ~90 lines to ~50 lines and isolates the menu logic for easier testing and modification.
  2. Add a Storage trait abstraction — Define a Storage trait in skymark-app with methods for read, write, list, and mtime. Implement it with StdStorage (wrapping std::fs) for the current codebase. This enables future swapping to encrypted storage, virtual file systems, or test doubles without changing Tauri commands.
  3. Implement incremental file watcher management — Replace the watch_paths/unwatch_paths pattern (which recreates the debouncer entirely) with a WatcherManager that adds and removes individual file watches. This eliminates the brief period where file changes are not detected during watcher swap and reduces resource churn.
  4. Add content-hash render caching in skymark-core — Cache render results keyed on a content hash (e.g., FNV-1a or xxhash) so that re-parsing is skipped when the document content has not changed since the last render. This reduces CPU usage on repeated edits and helps maintain the 50ms latency budget.
  5. Add a depth limit to vault scan — Cap the recursive directory traversal at a configurable depth (e.g., 20 levels) to prevent stack overflow on deeply nested directories. The 5000-file cap should be paired with a depth cap for defense in depth.
  6. Add retry logic to file watcher initialization — If watch_paths fails (e.g., file deleted between scan and watch), retry with exponential backoff (3 attempts, 1s / 2s / 4s intervals) before surfacing an error to the user. This handles transient file system state changes gracefully.
  7. Add a draft write mutex — Use a tokio Mutex (or std::sync::Mutex with a guard pattern) to serialize draft writes. This prevents the auto-save timer and explicit save from racing on the same draft file, ensuring the recovery system always has a consistent view of the latest draft.
  8. Add structured logging — Integrate a lightweight logging crate (e.g., tracing) with a file sink for the Rust backend and a console-based logger for the frontend. This enables diagnosing issues in production without requiring the user to open developer tools.

Migration Notes

Phase 1 (low effort, high impact): Extract menu construction from main.rs (Recommendation 1) and add a depth limit to vault scan (Recommendation 5). These are single-file changes with immediate benefits.

Phase 2 (moderate effort): Implement incremental file watcher management (Recommendation 3) and add a draft write mutex (Recommendation 7). These require changes to the watcher and draft modules but are self-contained.

Phase 3 (higher effort): Add a Storage trait abstraction (Recommendation 2) and content-hash render caching (Recommendation 4). These are architectural changes that touch multiple modules and should be planned as a dedicated effort.
