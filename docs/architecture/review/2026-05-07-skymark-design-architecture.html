Skymark Architecture Review

Cross-platform Tauri 2 desktop Markdown editor with a pure-Rust core that also targets WASM for a future web app.

Generated 2026-05-07 · Design Review

Executive Summary

Skymark is a fast, smart, lightweight Markdown editor built as a Tauri 2 desktop app on macOS, Linux, and Windows, with a pure-Rust skymark-core crate that also compiles to wasm32-unknown-unknown to keep a future web-app path open. The architecture is well-partitioned (core / host / frontend), enforces its layering invariant in CI, and uses a single canonical Markdown→HTML pipeline shared between live preview and HTML export. The most material gaps are non-functional: webview content security and untrusted-markdown sanitization, an unspecified save/auto-save/crash-recovery model, and a few under-specified contracts (AST↔DOM source-mapping, fuzzy-search index lifecycle, render cancellation).

Architecture

Strength
Separation of Concerns

Three cleanly partitioned hosts: skymark-core (logic, no I/O, no UI), skymark-app (Tauri-resident I/O and host APIs), frontend (presentation). The layering invariant — core has zero Tauri/JS deps — is explicit and CI-enforced via the wasm32-unknown-unknown build gate on every PR. That gate is unusually strong: the design refuses to drift architecturally without breaking CI.

Strength
Hexagonal / Ports & Adapters

The FormatConverter trait is a textbook port. Built-in adapters (HtmlExporter, DocxRsExporter) live in core; host-only adapters (WebviewPdfExporter) live in skymark-app and register into the same ConverterRegistry. The trait's bidirectional shape means import providers slot in symmetrically without churn.

Strength
Cohesion: single canonical render pipeline

The live preview and the HTML exporter both call skymark-core::render_html. Preview and export cannot drift — a high-cohesion choice that prevents a class of "looks right in preview, wrong in export" bugs that plague Markdown editors.

Concern
Source-map contract underspecified

Diff rendering depends on a stable mapping from AST blocks to DOM elements; scroll sync depends on per-block source-line ranges. The spec describes both but does not pin down the carrier of identity (block hash? structural path? source-range tuple?), how identity survives reordering, or how the frontend resolves cursor-line to a DOM block. This boundary is the kind that quietly produces drift and edge cases later.

Concern
Smart-paste image path crosses a boundary the diagram doesn't show

The frontend cannot write files directly under Tauri 2 without an exposed command. Saving a pasted image into .attachments/ implicitly requires a backend command that the spec doesn't enumerate. Adding an explicit "save attachment, return relative path" Tauri command keeps the file-I/O port clean and prevents the frontend from accreting platform-specific code.

Security

Strength
Local-first, no network surface in v1

No AuthN/AuthZ, no remote endpoints, no shared state. The OWASP attack surface collapses to local file-system access and webview content handling.

Risk
Untrusted Markdown → preview XSS

CommonMark allows raw HTML pass-through by default. Markdown files arrive from arbitrary sources (paste, import, synced folder). If the preview renders raw HTML or executes inline scripts, a malicious file gains JavaScript execution inside a webview that can call Tauri commands — exposing the file system. The spec does not name a sanitizer or a Content-Security-Policy. Required v1 controls: an HTML sanitizer (e.g., ammonia) at the core boundary, and a strict CSP on the preview frame disallowing inline scripts and remote resources except the explicit KaTeX/Mermaid/Prism assets.

Concern
Tauri 2 capabilities not enumerated

Tauri 2's security model relies on per-window capability files declaring exactly which commands and fs paths the frontend may access. Default-permissive misconfigurations are a common source of Tauri vulnerabilities. The spec should commit to an allowlisted command set and dynamic fs-scope rooted at the active vault (and the app config dir, and a temp scratch path) — not the whole disk.

Concern
Smart-paste image input validation

Writing arbitrary clipboard image data to disk under a hash-named path needs explicit defenses: extension allowlist (png/jpg/gif/webp/svg-with-care), magic-byte sniffing to confirm the type matches the extension, a size cap, and SVG handling that strips embedded scripts (or refuses SVG entirely). Without these, a crafted clipboard payload becomes a write-anywhere primitive.

Concern
v1.1 Pandoc download supply-chain integrity

Architectured-not-shipped, but worth pinning now: any "prompts a one-time download" of Pandoc must require HTTPS + checksum/signature verification of the binary against a hardcoded expected hash per platform/version. Otherwise it's a download-and-execute vector. Note now so v1.1 inherits the requirement.

Scalability

Strength
Explicit, CI-enforced performance budgets

The 50ms preview budget, 1.5s cold start, and 10k-word fixture are baked into CI as a perf test. That's the most disciplined scaling guardrail a desktop editor can have — performance regressions are caught at PR time, not at user time.

Strength
Diff rendering + lazy module loading

Per-block diff rendering keeps the keystroke loop sub-budget on large documents. Lazy-loading Mermaid and Prism language packs keeps cold start light when their features are unused.

Concern
Render-pipeline cancellation/coalescing not specified

If a single render call exceeds the debounce window (large paste, deep nested lists, pathological math), a newer keystroke can arrive while an older render is in flight. Without a cancel-stale / coalesce-latest rule, the UI can apply older-than-current results. Standard mitigation: tag each invoke with a monotonic id; the controller commits only the latest id.

Concern
Fuzzy-search index lifecycle underspecified

Settings caps index size, but the spec does not state when the index is built (eager on vault open, incremental on file events, persisted across sessions), the eviction policy at the cap, or the user-visible behavior when a vault exceeds it. For vaults of 5k+ files this is the difference between "instant search" and "two-second hang on first search."

Concern
Vault file-watcher fan-out

Watching a large vault (10k+ files) on macOS can hit FSEvents thresholds and burn battery; on Linux, inotify watch limits apply. The spec mentions notify but doesn't address debouncing, batching, or graceful degradation when watcher capacity is exhausted. Add a debounce window (e.g., 200ms) and a "watch root, not all files" strategy where the platform supports it.

Concern
Documents above ~50k words

Spec acknowledges this as out of scope. Acceptable disclosure for v1, but the diff-render heuristic is the limiter; a virtualized preview is the v1.1 escape hatch worth flagging now.

Reliability

Strength
External-edit handling defined

The file watcher → prompt-or-merge flow addresses a real reliability concern for desktop editors (silent overwrite of user changes). Most editors get this wrong; calling it out in the design is a strength.

Strength
Golden-file conversion tests

Each converter has golden-file regression tests (with canonical XML diff for DOCX). Export fidelity is regression-protected at PR time — a strong reliability commitment.

Risk
Save / auto-save / crash recovery not specified

The spec mentions "passive scan on save" and "save settings," but never names the save model: explicit cmd-S only? auto-save on idle? on focus loss? What persists when the OS kills the app mid-edit? For an editor, losing 10 minutes of typing is the canonical failure users blame the product for. Recommended: periodic write-to-temp for unsaved buffers, with recovery on next launch, plus a clear save-on-close prompt for dirty buffers.

Concern
DOCX out-of-subset UX undefined

The DOCX exporter ships a defined subset; the spec says the export dialog "surfaces this," but does not specify whether out-of-subset content (footnotes, math) blocks export, warns and proceeds with loss, or silently drops. Pick one; "warn-and-proceed-with-explicit-list" is the user-respecting default.

Concern
Webview-PDF cross-platform fidelity

Print-to-PDF output depends on the platform webview (WebKit / WebView2 / WebKitGTK). Pagination, font rendering, and CSS support differ. Acceptable for v1 but worth disclosing in docs and golden-testing per platform — otherwise user reports of "PDF differs between Mac and Windows" become support load.

Recommendations

  1. Pin webview content security in v1 — choose an HTML sanitizer at the core boundary (e.g., ammonia), decide whether raw HTML in Markdown is allowed (default: stripped), and apply a strict Content-Security-Policy on the preview frame disallowing inline scripts and remote loads outside an explicit allowlist.
  2. Specify the save / auto-save / crash-recovery model — add a Persistence section: when buffers are written to disk, where unsaved drafts land between writes, and the recovery flow on next launch.
  3. Lock down Tauri capabilities and fs scope — enumerate the exact command allowlist; scope fs writes to the active vault, app config dir, and a controlled scratch path; document the dynamic-scope mechanism so vault-switch updates capabilities atomically.
  4. Pin the AST↔DOM source-map contract — name the carrier of block identity, define how identity survives reorders, specify how cursor-line resolves to a DOM block for scroll sync.
  5. Add render cancellation / coalescing — monotonic-id tagging on invoke; the preview controller commits only the most recent id and discards stale results.
  6. Define the fuzzy-search index lifecycle — build strategy (eager / incremental), persistence, eviction at the cap, and degraded-mode behavior when a vault exceeds the cap.
  7. Specify file-watcher debouncing and degraded mode — debounce window (suggest 200ms), watch-root strategy where supported, and behavior when platform watcher capacity is exhausted.
  8. Add an explicit "save attachment" Tauri command — keeps the smart-paste image path through a single audited port; include extension allowlist, magic-byte sniff, size cap, and SVG-script-strip (or SVG-refuse).
  9. Define DOCX out-of-subset UX — warn-and-proceed with an explicit list of dropped/lossy elements is the recommended default; commit to it in the spec so users aren't surprised.
  10. Pre-commit v1.1 Pandoc supply-chain controls — HTTPS + per-platform hardcoded checksum verification before any downloaded binary is executed. Note in the spec now so v1.1 inherits the constraint.
