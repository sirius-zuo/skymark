skymark — Format Toolbar Design Review

Architecture review of the format toolbar feature for the Skymark Markdown editor.

Generated 2026-05-10 · Design Review

Executive Summary

The format toolbar adds a fixed strip of 16 formatting buttons above the CodeMirror editor in the Skymark Tauri desktop app. Architecturally, the design is clean: a new toolbar.ts module handles DOM construction and event wiring, while three new exported functions in editor.ts (wrapSelection, toggleLinePrefix, insertTemplate) encapsulate all CodeMirror transaction logic. The design is well-suited to the feature's scope — stateless, synchronous, and zero external dependencies — with one minor reliability concern around a missing null guard on element lookup.

Architecture

Strength
Separation of Concerns

toolbar.ts owns only DOM construction and event wiring. editor.ts owns only CodeMirror transaction logic. Neither leaks into the other's responsibility, and main.ts is the sole wiring point.

Strength
Layered Architecture

Dependencies flow in one direction: main.ts → toolbar.ts → editor.ts → CodeMirror. No reverse dependencies or circular imports.

Strength
Minimal API Surface

Only three functions are added to editor.ts's public API, covering the full range of formatting operations (wrap, prefix-toggle, insert). The three functions are orthogonal and composable — link button is simply wrapSelection("[", "](url)").

Concern
editor.ts God Module Risk

editor.ts is the accumulation point for all editor-related logic. The spec adds three more exported functions without addressing how large the file currently is. As the project continues to grow, this file risks becoming a god module. The spec's "Out of Scope" list keeps v1 small, but the risk should be tracked.

Security

Strength
No Injection Risk

All text transformations go through CodeMirror's transaction API (view.dispatch). No use of innerHTML, eval, or raw DOM string injection. Template strings are hardcoded constants, not user-supplied.

Strength
No Auth or Secrets Surface

This is a local desktop feature with no network calls, no credentials, and no external service integration. The OWASP Top 10 categories that involve authentication, secrets, or data exposure are not applicable.

Scalability

Strength
Stateless Design

toolbar.ts carries no mutable state of its own. All document and cursor state lives in CodeMirror's EditorView. The toolbar is effectively a pure function of user interaction events.

Strength
Bounded Operations

All button operations are O(lines in selection) at worst — appropriate for a text editor. No unbounded loops, no network round-trips, no async work. Performance is not a concern.

Reliability

Strength
Focus Preservation

All buttons use mousedown + e.preventDefault(), which correctly prevents the editor losing focus on click. This is the most common reliability failure in toolbar implementations and is explicitly addressed in the spec.

Concern
Non-null Assertion on Element Lookup

main.ts uses document.getElementById("format-toolbar")! with a non-null assertion. If the element is ever absent (e.g., HTML and JS get out of sync during a build), this throws a runtime exception rather than logging a graceful error. A null check with a console warning would be more resilient.

Recommendations

  1. Guard the createToolbar call — Replace the ! non-null assertion in main.ts with a runtime null check: const el = document.getElementById("format-toolbar"); if (!el) { console.error("format-toolbar element not found"); return; } createToolbar(el, editor.view);. This prevents a silent crash if HTML and JS ever diverge.
  2. Track editor.ts size — After implementation, note the line count of editor.ts. If it exceeds ~300 lines, plan to extract formatting functions into editor-format.ts in the next editing pass. The spec correctly defers this, but it should be on the radar.
